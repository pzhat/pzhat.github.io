CBJS Java Misconfiguration Challenge

Statement Viewer 1

Đây là Application có chức năng là upload file chỉ có thể upload file pdf hoặc là file txt bên cạnh đó là chức năng view statement để xem lại những gì mà mình đã upload lên.

Sau khi upload file lên ta có thể download file về ở đây với mỗi file được up lên đều có một file path riêng để lưu lại file đó. Vì bài này nó cho sẵn source là white box nên ta sẽ phân tích code.

Code Breakdown

Vì đây là lab misconfig nên mình sẽ đi vào các nơi gây ra lỗi cụ thể ở đây là các cửa ngõ mà dev để quên khi test web app từ đó attacker có thể lợi dụng chúng để có thể đi sâu vào hệ thống và chiếm quyền điều khiển.

Đến với file config là context.xml ta có thể thấy rằng ở đây có một setting đó là privileged="true" cấu hình này cho phép :

  • Cho phép web app truy cập các internal API của Tomcat

Có thể:

  • Load class nội bộ của Tomcat

  • Gọi các Servlet/Valve nhạy cảm

  • Truy cập JMX, Manager API (nếu misconfig thêm).

Và bình thường cấu hình này sẽ được dùng cho /manager và /host-manager nơi mà giúp quản lý apache tomcat server.

Nếu như có mỗi cấu hình đó thì cũng sẽ khó để khai thác thêm nhưng ở đây nó còn có thêm cả đoạn.

Ở đây nó được config rằng tất cả IP đều sẽ truy cập được vào các nơi nội bộ và nếu attacker nắm trong tay tài khoản nào đó như là tài khoản admin tomcat chẳng hạn thì sẽ có thể vào được trang quản lý web.

Vậy là ta có mắt xích đầu tiên là context.xml sau đó là đến với tomcat-users.xml đây là một file cấu hình cho apache tomcat.

Và bên trong file này ta có được mắt xích thứ 2 của bài, ở đây anh dev đã để quên mất một tài khoản của user spike và bên cạnh đó tài khoản này được config các role rất cao của admin từ đây ta hoàn toàn có thể lợi dụng chain này để tiến hành truy cập vào tomcat manager.

Exploit

Ta truy cập vào /manager/html sau đó nhập username và password đã khai thác được từ đây ta đã thành công truy cập được vào trang quản lý của apache tomcat.

Tại tomcat manager ta để ý rằng nó tồn tại chỗ để upload file WAR để deploy lên tại đây ta hoàn toàn có thể tiến hành upload 1 cái shell ở đây để có thể thực hiện điều khiển server (RCE).

Tại đây mình sẽ sử dụng một tool đó là godofwar để tạo file WAR shell tham khảo thêm tại :

  • https://medium.com/defmax/rce-via-war-upload-in-tomcat-using-path-traversal-e0f11898016e
  • https://medium.com/@mingihongkim/exploiting-java-portlets-with-a-malicious-war-file-to-gain-a-reverse-shell-2504909f71c1

Sau khi upload và deploy ta chỉ việc truy cập đến file shell và thành công RCE được web server.

Statement Viewer 2

Đến với level 2 thì nó vẫn không có gì khác biệt chức năng chính của web app này sẽ vẫn giống như cũ code hoạt động cả cách nó xử lý vẫn sẽ y hệt nên ta sẽ đi tìm nơi bị misconfig để khai thác mục tiêu ở đây cũng vẫn là RCE.

Sau khi thực hiện deploy web app lên thì ta để ý rằng có một đoạn trong log nó start ajp-nio-8009 bên cạnh đó bên trong file docker-compose có đoạn expose port 8009.

Vậy cái nơi mà port 8009 đó là cái gì? Theo tìm hiểu thì đó là Apache Jserv Protocol (AJP) nó là một giao thức nhị phân được thiết kế để proxy request từ front-end web server như là Apache HTTPd tới back-end server container (Tomcat).

Và vấn để ở đây là nếu misconfig hay nghĩa là AJP bị expose ra ngoài internet thì nó sẽ thành nơi mà attacker có thể tương tác với port của AJP và thao túng nó.

Tomcat dựa vào request được chuyển từ AJP để truy cập tài nguyên webapp bằng API gốc của servlet container - nếu request đó bị crafted đặc biệt, sẽ dẫn tới file read/include.

Vì thế: nếu app có upload và file nằm trong webapp (hoặc attacker control content webapp) thì có thể dẫn đến RCE mà ở đây trong web này nó là một web app với chức năng upload file và download nội dung upload về thì nó là nơi có thể thực hiện RCE.

Sau khi research thì đây chính là combo tạo nên Ghostcat(CVE-2020-1938).

Sau khi thử truy cập đến port 8009 thì có vẻ như nó chẳng có cái gì hoặc có thể nó không cho phép ta tương tác UI nên ta sẽ tìm hướng khai thác theo kiểu khác.

Sau khi research mình tìm được quả tool đê test : https://github.com/00theway/Ghostcat-CNVD-2020-10487

Ở đây mình test vào chall thật luôn ko test local nữa và thành công đọc được file bên trong bây giờ ta sẽ tận dụng nó để có thể thực hiện RCE.

Tại đây ta tiến hành upload một file shell.txt sau đó thực hiện sử dụng tool python có sẵn ở đây dùng lệnh :

Ở đây nhớ là phải thay đổi folder có chứa shell vì folder nó là userID.

Sau khi chạy ta đã thành công RCE bằng cách lợi dụng port 8009 bị expose của server đến đây tự cat flag là xong.

CyberSoc

Đây là một trang quản lý SOC về luồng xử lý thì nó cũng chưa có gì hấp dẫn vì đây là lab về misconfig nên mình sẽ đi vào các đoạn sink misconfiguration là chủ yếu.

Check souce code thì ta có thể thấy web app này xài Spring Boot và đặc biệt nó có xài Actuator.

Ở đây ta biết được actuator nó là một nơi để mình quản lý project và chính nơi này trong tài liệu của Spring cũng nói rằng nơi đây thường xuyên bị lộ những thông tin bí mật nên khả năng cao ta sẽ khai thác từ các endpoint của actuator.

Sau khi truy cập vào đọc code thì ta để ý ở file SecurityConfig bị leak rất nhiều endpoint của admin lẫn actuator và cái bị sai ở đây là dev đã config cho mọi người đều có thể truy cập đến các tài nguyên của API /api/actuator/... và đây là nơi ta có thể lợi dụng để khai thác thêm thông tin.

Ở đoạn application.properties cũng đã config rằng nó expose endpoits actuator như là info,health,env,metrics,logfile,mappings,configprops nên bất kỳ user nào quyền nào cũng có thể vào được thậm chí không cần đăng nhập cũng được.

Tiếp theo đó tại data.sql ta có thể thấy rằng nó INSERT vào 2 user đó là staff.cybersoc có role là USER và admin với role là ADMIN tuy ở đây nó leak password nhưng đã được mã hoá bằng Bcrypt nó là mã hoá 1 chiều nên không decode ra được nên ta sẽ phải tìm cách khác để đăng nhập được vào.

Sau khi truy cập vào /api/actuator/mappings nó leak cho mình một đống endpoint cùng 1 vài config có thể lợi dụng nhưng cũng không cần lắm vì nó leak trong source rồi.

Sau khi truy cập vào /api/actuator/logfile ở đây có đoạn :

Vậy là ở đây mình đã tìm được một token bị leak ra và cái token này đáng lẽ nên bị xoá nhưng nó vẫn tồn tại bây giờ mình sẽ thử decode cái token này ra để xem nó là token của user nào.

Vậy là ta kết luận rằng đây là token của user staff.cybersoc và role của nó là USER bây giờ ta sẽ inject vào để truy cập vào /dashboard.

Trong JwtAuthorizationFilter ta có thể thấy được chỗ cookie có header là Authorization và bắt đầu với Bearer.

Thành công đăng nhập được vào dashboard và lấy được flag đầu tiên.

Truy cập vào /api/actuator/metrics sau khi vào đây thì có vẻ như không có gì mấy để khai thác nhiều nên ta sẽ đi đến enpoint khác để tìm kiếm thêm.

Truy cập vào 2 API endpoint là /admin/console và /api/actuator/configprops cả 2 đều trả về response là 403 là bị block có lẽ chính là vì nó bị config chỉ có admin được truy cập vào đây. Vậy bây giờ ta sẽ phải tìm hướng đi để leo được lên user có ROLE là admin.

Tại đây sau khi truy cập vào được /api/actuator/env ta đã tìm được một điểm misconfig lớn là nó allow user khác đọc được env điều này đã khiến cho SECRET_KEY của JWT đã bị leak ra, với key này ta có thể mod lại JWT và biến mình thành admin.

Ngon rồi cái token đã valid bây giờ mình sẽ inject thử.

Đầu tư vào biến token của user staff.cybersoc từ role USER lên ADMIN ngon luôn rồi, bây giờ truy cập vào H2 console response trả về 302 có vẻ như ta đã được redirect đến trang console.

Thành công truy cập tới trang console nhưng vấn đề là bây giờ thiếu mất đi password để có thể đăng nhập nên ta sẽ mò tiếp trong actuator xem nó có leak password không.

Thành công lấy được password của H2 db đăng nhập thành công được user nhờ vào API /api/actuator/configprops đã leak ra password.

Sau khi đăng nhập thì có một cái bảng tên là flag bây giờ chi việc chạy SQL query để đọc nội dung trong bảng flag.

Thành công lấy được thông tin trong bảng cụ thể ở đây là bảng FLAG. Bây giờ tuy đã đăng nhập vào được H2 Database bây giờ mình sẽ tìm cách leo quyền để có thể thực hiện H2 to RCE.

Sau khi research một lúc thì vô tình tìm được 1 nguồn hướng dẫn leo quyền ở H2 tham khảo ở : https://exp10it.io/posts/h2-rce-in-jre-17/

Sử dụng payload để thực hiện sử dụng shell với câu lệnh ls -la sau đó đưa output ra file output.txt.

Sau đó dùng read file để có thể đọc được output sau khi thực thi shell.

Thành công RCE và đọc được file flag cuối cũng và kết thúc lab misconfig.
